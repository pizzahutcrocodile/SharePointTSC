Inheritance Tutorial

First, you must think about how your folders are related to one another but also keep in mind to group them based on Permission Groups.

In SharePoint, access to websites, lists, folders, and list items is controlled through a role-based membership system by which users are assigned to roles that authorize their access to SharePoint objects.

In the list below, you will see the Permission Level name and right below that a list of authorization powers. For example, in the sixth number of the list the Permission Level "Read" grants the authority to view pages, list items, and download documents to whatever Permission Group "Read" is assigned to.

SharePoint has Default Permission Levels:

  1. Full Control
    • Has Full Control
  2. Design
    • Can view, add, update, delete, approve, and customize.
  3. Edit
    • Can add, edit and delete lists; can view, add, update and delete
  4. Contribute
    • Can view, add, update, and delete list items and documents.
  5. Review
    • Can view and review list items and documents
  6. Read
    • Can view pages and list items and download documents.

Authority is added to Permission Levels, Permission Levels are added to Permission Groups by Group Owner, Group Owner creates a Permission Group and then adds Users to the permission group (a user or a permission group can be assigned as Group Owner "Grant Permissions"), and finally Permission Groups are added to Folders.

All folders and files inside of the Parent folder will inherit the permission group, these folders are called Child folders.

For example, if all the folders are put inside a "Site Folder" and the Site Folder is assigned the "Read Only" permission group, that was assigned "Read" permission level, which means that all members inside of the Read Only group are authorized to view, list items and download documents as it is stated on the list above, it means that all folders inside of the "Site" folder (child folders) inherit the Read Only permission group.

Inside of the "Site Folder" there could be other folders that could be assigned more exclusive permission groups with permission levels like "Contributor." There might be an instance that a file or folder holds sensitive information that needs to be excluded from the Read Only Permission Group, then one must manually stop the inheritance in said folder (see "Remove a Group"). This can be done to all permission groups excluding the ones with the permission level of "Full Control".

A single user can belong to multiple groups. For example, if Sam is in both Read only and Contribute group, which has contribute level Permissions, and a folder has given both groups access then Juan is going to be given authority from both groups.

Because of this the initial Folder, which in this case is called "Team Site", usually has the permission level of Read. It is good practice to put all users in a group with Read Permission Level since it is the least confidential. The more exclusive Permission Groups should be more selective in who gets granted membership.

Let's say there are three Users named Juan, Sam and Jerry. Juan belongs to the Read Only Group, Sam belongs to the Contribute Group and Read Only Group, and Jerry is a member of the Owner Group, which has Full Control permission level.

Juan belongs to Team A, Sam to Team B and Jerry is the manager. Jerry wants to stop Juan from reading files in Team B and Sam From Reading files from Team A. Due to "Inheritance" all subsequent folders and files inherit permission levels from the Parent folder which is the Read Only permission group.

Jerry has to go to the Folders themselves and manually remove the Read only Permission Group and add a custom Permission Group made for Team A and Team B. Jerry named them Team A and Team B for clarity. Those Permission Groups are given Read permission Level and that's it.

Jerry then has to add the "Contribute Permission Group" to the Team B Folder (see "Add a Group") so Sam can have Contribute level permission. Since Jerry added Contribute Permission Group to Team B all subsequent folders and files inherits Contribute Level Permissions. Because Jerry has Full Control Permission Level he can always see and edit what he likes regardless of the Permission Group explicitly added to the folder.

Because Juan is in the Read Only Group and Team A Group and not in the contribute permission group like Sam he can't add, update or delete like she can and even if he was Jerry has not added the Contribute Group to the Team A folder so Contribute Group is not allowed on to the Team A folder.
